# Personal Mastery Program

Subtitle: A self-directed graduate-level curriculum tailored to your goals.

## Mission

Build graduate-level capability in public, starting with CS/AI Systems, so the missing Master's credential is offset by visible skill, strong fundamentals, and serious artifacts.

## Current positioning

Default identity for 2026: AI-native software engineer building an MSc-level CS/AI curriculum in public.

A formal Master's remains an optionality track, not the center of the plan. Reevaluate applications only when the proof base is materially stronger or when a job/startup opportunity changes the decision.

## Current track

CS/AI Systems: computer science fundamentals, backend systems, AI/ML, LLM engineering, production engineering, research reading, and technical writing.

Primary target: AI-native backend/product systems engineer.

## Principles

- Capability first, legibility second.
- Agents are teachers and examiners; sources are authorities.
- Public proof matters because there is no automatic credential.
- Public proof must not distort module goals.
- Modules should be small enough to complete in weeks, not semesters.
- Interleave tracks, but give each cycle one dominant focus.
- Use source-backed teaching, artifact-backed assessment, and Git-backed reviews.

## Weekly budget

Planned effort: 15-20 focused hours per week.

Default allocation during CS/AI Systems cycles:

- 50% dominant technical track
- 25% supporting CS fundamentals
- 15% ML/LLM under the hood
- 10% career, writing, portfolio, and public proof

## Evidence model

Every module must produce evidence. Not every module must produce polished public proof.

- Evidence: exercises, notes, oral exam transcript, reference sheet, mini-output.
- Validated artifact: checked implementation, benchmark, essay, reproduction, or architecture note.
- Public proof: polished artifact suitable for recruiters, peers, future learners, or admissions reviewers.

## Assessment model

Each module declares its own feedback and assessment mix. The shared menu includes:

- closed-book recall
- implementation task
- debugging task
- design review
- oral defense
- written synthesis
- benchmark or eval
- paper reproduction
- external exercise set
- project integration
- public artifact review

Most technical modules should include at least one retrieval component and one application component.

## First milestone

Within the first 8-12 weeks, produce a portfolio-grade AI backend system with public evidence around architecture, retrieval or memory, agent/tool boundaries, async workflows, evals, observability, deployment, and written tradeoffs.
