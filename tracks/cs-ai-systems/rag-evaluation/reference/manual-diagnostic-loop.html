Manual RAG Diagnostic Loop

Use this before metrics. One failed answer should become one specific engineering action.

The Loop

1. Expected facts

What exact claims should a good answer contain?

2. Expected sources

Which files contain those claims?

3. Retrieved chunks

Did the system retrieve the source material?

4. Actual answer

Did the model use the retrieved material correctly?

5. Failure label

Which part failed?

6. Next action

What should you change first?

Diagnosis Table

Observation Label Next action
The needed fact is not in the corpus. missing_source Add or update source docs before tuning retrieval.
The corpus has the fact, but retrieval did not fetch it. retrieval_miss Change chunking, indexing, query rewrite, filters, or top-k.
The right fact was retrieved with too much irrelevant context. retrieval_noise Adjust chunk size, reranking, filters, or context budget.
The right context was retrieved, but the answer changed or invented facts. generation_drift Change prompt, grounding rule, model, or refusal behavior.
The answer is grounded but misses required facts. answer_incomplete Clarify expected facts or prompt for complete coverage.
The question or expected answer is too vague to judge. dataset_ambiguous Repair the eval row before judging the system.

This is the usefulness of eval rows: they turn "the assistant was bad" into "fix retrieval", "fix generation", "add a source", or "repair the test".

Where Metrics Enter Later

Metrics such as context recall, context precision, answer relevancy, and faithfulness are automation around this same loop. Use the glossary for terms and the DeepEval RAG evaluation guide for how tool metrics separate retriever and generator behavior.

Do not start by optimizing scores. Start by learning what a failed case is telling you.
